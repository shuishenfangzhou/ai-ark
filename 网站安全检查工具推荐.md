**3\. 网络层主动侦察与扫描策略**

在完成了被动信息收集后，进入主动扫描阶段。对于 IP 120.26.35.49，我们不能仅仅测试 8080 端口，而必须对整个网络协议栈进行探测，以了解 8080 端口存在的上下文环境。

### **3.1 Nmap：全端口扫描与协议分析**

**Nmap (Network Mapper)** 是进行此类彻底检查的基石工具 8。为了达到“彻底”的标准，简单的 ping 扫描或默认的 top 1000 端口扫描是远远不够的。必须针对 TCP 协议的握手过程进行定制化探测。

#### **3.1.1 扫描深度与范围**

建议的 Nmap 扫描策略应包含全端口（1-65535）扫描，以发现可能与 8080 端口服务相关联的其他隐蔽端口。例如，Apache Tomcat 通常在 8080 开放 HTTP 服务的同时，会在 8005 开放关闭（Shutdown）端口，在 8009 开放 AJP（Apache JServ Protocol）端口。如果仅检查 8080 而忽略了 8009，可能会遗漏严重的“Ghostcat”漏洞 10。

**推荐的 Nmap 深度扫描命令逻辑：**

该命令不仅探测端口，还调用脚本引擎（NSE）进行漏洞探测。

Bash

nmap \-p 1-65535 \-sV \-sC \-O \--version-all \--script "default,http-vuln\*,http-config-backup,vuln" \-T4 120.26.35.49

* **\-p 1-65535：** 确保不遗漏任何高位端口。  
* **\-sV (Version Detection)：** 这一点至关重要。Nmap 将通过发送特定的探针数据包并分析响应指纹，来精确判断运行在 8080 端口的软件及其具体版本号（例如，“Apache Tomcat 9.0.30”）。精确的版本号是后续查找 CVE（通用漏洞披露）数据库的关键。  
* **\-sC (Script Scan)：** 启用 Nmap 默认的脚本库。这些 Lua 脚本可以自动检测常见的 HTTP 错误配置、默认文件以及 SSL 证书信息。  
* **\-O (OS Fingerprinting)：** 通过分析 TCP/IP 协议栈的指纹（如 TTL 值、窗口大小等）来推断服务器的操作系统（Linux Kernel 版本或 Windows Server）。

#### **3.1.2 针对防火墙与 IDS 的规避**

考虑到目标位于阿里云，可能会有云盾（Aliyun Aegis）或 Web 应用防火墙（WAF）进行防护。如果高强度的扫描导致连接被阻断（Connection Reset），需要调整扫描策略：

* **分片扫描 (-f)：** 将 TCP 头分片，试图绕过简单的包过滤规则。  
* **慢速扫描 (-T2)：** 降低发包速率，模拟正常流量，避免触发流量异常告警。  
* **源端口伪造 (--source-port 53)：** 某些防火墙规则可能允许来自 DNS 端口（UDP 53）的流量通过。

### **3.2 辅助扫描工具**

除了 Nmap，为了确保结果的准确性，还应交叉使用其他工具：

* **Masscan：** 如果需要对整个 C 段（120.26.35.0/24）进行存活探测，Masscan 的异步发包机制速度极快 11。但针对单个 IP 的深度检查，其精度不如 Nmap。  
* **RustScan：** 现代化的端口扫描器，利用 Rust 语言的高并发特性快速发现开放端口，然后自动调用 Nmap 进行深层分析，提高了检查效率 12。  
* **Netcat (nc)：** 用于手工验证。当 Nmap 报告 8080 开放但无法识别服务时，使用 nc \-v 120.26.35.49 8080 手动发送 HTTP 请求头（如 GET / HTTP/1.0），观察原始回显，往往能发现被自动化工具忽略的细节。

**4\. 应用层漏洞评估（VA）与深度检测**

确认了 8080 端口开放且运行了特定的 HTTP 服务后，检查工作进入核心阶段：漏洞评估。这一阶段的目标是发现软件缺陷、配置错误和逻辑漏洞。我们将针对三种最可能的场景（Tomcat、Spring Boot、通用 Web 服务）进行深度剖析。

### **4.1 场景 A：Apache Tomcat 深度检查**

如果 Nmap 识别出 8080 端口运行的是 Apache Tomcat，那么这是攻击者最喜欢的靶标之一。Tomcat 的默认配置和历史漏洞使其成为高危资产。

#### **4.1.1 Ghostcat 幽灵猫漏洞 (CVE-2020-1938)**

这是一个存在于 Tomcat AJP 协议（默认端口 8009）中的文件包含漏洞 10。虽然用户只询问了 8080，但“彻底检查”必须验证 8009。

* **检测原理：** Tomcat 默认开启 AJP 连接器以便与 Apache HTTPD 通信。如果该端口暴露在公网且未升级补丁，攻击者构造恶意的 AJP 数据包，可以读取 Web 目录下的任意文件（如 WEB-INF/web.xml）甚至源代码。  
* **检查工具：**  
  * **Nmap 脚本：** nmap \-p 8009 \--script ajp-ghostcat 120.26.35.49  
  * **Metasploit 模块：** auxiliary/admin/http/tomcat\_ghostcat  
* **彻底性验证：** 即使 8009 端口在防火墙层面被阻断，也应检查 Tomcat 版本是否低于 9.0.31、8.5.51 或 7.0.100。如果是，则潜在风险依然存在。

#### **4.1.2 Tomcat Manager 弱口令与部署漏洞**

Tomcat 自带一个管理后台，通常位于 /manager/html。

* **风险点：** 管理员常为了方便设置弱口令（如 admin/admin, tomcat/tomcat），或者在 tomcat-users.xml 中保留了默认示例用户 14。  
* **攻击后果：** 一旦登录 Manager 后台，攻击者可以上传恶意的 WAR 包（Web Application Archive），Tomcat 会自动解压并部署该包，从而导致远程代码执行（RCE），直接控制服务器。  
* **检查工具：**  
  * **Hydra：** 用于暴力破解认证。命令示例：hydra \-L users.txt \-P passwords.txt \-s 8080 \-f 120.26.35.49 http-get /manager/html  
  * **Metasploit：** 模块 scanner/http/tomcat\_mgr\_login 16。  
* **彻底性标准：** 检查不仅要看能否登录，还要看 /manager 路径是否对公网 IP 开放。最佳实践是限制该路径仅允许 127.0.0.1 访问。

#### **4.1.3 HTTP PUT 方法任意文件上传 (CVE-2017-12615)**

在某些错误配置（readonly=false）的 Tomcat 版本中，攻击者可以通过 HTTP PUT 请求直接向服务器上传 JSP Webshell。

* **检测方法：** 向服务器发送 OPTIONS / HTTP/1.1 请求，查看响应头中的 Allow 字段是否包含 PUT 或 DELETE。

### **4.2 场景 B：Spring Boot Actuator 深度检查**

现代 Java 应用常基于 Spring Boot 框架构建，该框架默认在 8080 端口启动。

#### **4.2.1 Actuator 端点泄露**

Spring Boot 提供了 Actuator 模块用于监控应用健康状态。如果配置不当，会暴露敏感信息 17。

* **关键端点检查列表：**  
  * /actuator/env 或 /env：显示环境变量，可能包含数据库密码、AWS/阿里云 AK/SK 密钥。  
  * /actuator/heapdump 或 /heapdump：下载 JVM 内存堆转储文件。这是极其危险的漏洞。攻击者可以使用 Eclipse MAT 工具分析该文件，提取出内存中缓存的用户 Session ID、明文密码等。  
  * /actuator/mappings：列出所有 API 路由，帮助攻击者梳理业务逻辑。  
* **检查工具：**  
  * **手动 Curl：** curl http://120.26.35.49:8080/actuator/health。如果返回 JSON 格式的 {"status":"UP"}，则说明 Actuator 已启用。  
  * **自动化工具：** 专门的 Spring Boot 扫描脚本（如 GitHub 上的 SpringBoot-Scan）可以批量检测这些端点。

#### **4.2.2 SpEL 表达式注入**

如果 Spring Boot 应用在处理用户输入时使用了 Spring Expression Language (SpEL) 且未做过滤，可能导致 RCE。这通常需要结合具体的代码审计，但在黑盒测试中，可以尝试在输入框注入 SpEL 载荷进行 Fuzzing。

### **4.3 场景 C：Nginx 反向代理配置错误**

如果 8080 端口运行的是 Nginx（作为反向代理指向后端应用），配置错误是主要风险源 19。

#### **4.3.1 CRLF 注入**

如果 Nginx 配置了不安全的跳转（如使用 $uri 且未解码），攻击者可能注入换行符（CRLF），导致 HTTP 响应拆分或缓存投毒。

#### **4.3.2 目录遍历（Alias Traversal）**

Nginx 的 alias 指令如果未以斜杠结尾，可能允许攻击者穿越目录。

* **示例：** 配置 location /i { alias /data/w3/images; }。  
* **攻击：** 请求 /i../app/config.py 可能读取到 /data/w3/app/config.py。  
* **检查工具：** **Nikto** 是发现此类服务器配置错误的利器。

**5\. 工具生态系统与实操指南**

为了执行上述检查，用户需要构建一个包含多种工具的检测环境。以下是各类工具的详细实操指南与对比分析。

### **5.1 综合漏洞扫描器 (Vulnerability Scanners)**

这类工具就像“全身体检仪”，能自动化覆盖上述的大部分检查点。

#### **5.1.1 Nessus Professional (行业标准)**

* **适用性：** 商业软件，拥有最全的漏洞库 21。  
* **针对 120.26.35.49 的配置：**  
  1. 创建一个 "Advanced Scan" 策略。  
  2. 在 "Discovery" 设置中，将端口扫描范围设为 "All ports"。  
  3. 启用 "Web Application Scanning" 模块，目标 URL 填入 http://120.26.35.49:8080/。  
  4. **优势：** Nessus 对 Tomcat 版本漏洞的识别非常精准，误报率低。它可以直接告诉用户：“该服务器运行的 Tomcat 8.5.4 存在 CVE-XXXX 漏洞，建议升级到 8.5.50”。

#### **5.1.2 OpenVAS / Greenbone (开源选择)**

* **适用性：** 免费开源，适合预算有限的彻底检查 23。  
* **操作：**  
  1. 安装 GVM (Greenbone Vulnerability Management)。  
  2. 同步最新的 NVTs (Network Vulnerability Tests) 库。  
  3. 配置 Target 为 120.26.35.49，Port List 选择 "All TCP"。  
  4. **注意：** OpenVAS 的 Web 界面配置较为繁琐，扫描速度较慢，但其报告非常详细。

### **5.2 Web 应用程序安全扫描器 (DAST)**

这类工具专注于 HTTP 协议层面的逻辑漏洞，如 SQL 注入、XSS 等。

#### **5.2.1 OWASP ZAP (Zed Attack Proxy)**

* **推荐理由：** 免费、开源、功能强大，由 OWASP 维护，非常适合检查单体 Web 应用 23。  
* **彻底检查步骤：**  
  1. **Spider（爬虫）：** 配置 ZAP 对 http://120.26.35.49:8080 进行爬取，抓取所有可访问的链接。  
  2. **Forced Browse（强制浏览）：** 使用 ZAP 内置的字典（DirBuster wordlists）探测隐藏目录（如 /backup, /db, /admin）。  
  3. **Active Scan（主动扫描）：** 对抓取到的每个 URL 进行攻击模拟。ZAP 会尝试在参数中注入 SQL 语句、脚本标签等。  
  4. **分析：** 查看 "Alerts" 标签页，ZAP 会将风险按 High/Medium/Low 分类。

#### **5.2.2 Burp Suite Community/Pro**

* **推荐理由：** 渗透测试人员的首选工具，适合手动深度挖掘 24。  
* **操作：**  
  1. 设置浏览器代理指向 Burp。  
  2. 手动访问网站，点击各个功能点。  
  3. 将关键请求发送到 "Repeater" 模块，手动修改参数进行测试。  
  4. 如果是 Pro 版，可以使用 "Scanner" 自动发现漏洞。

### **5.3 专用工具与脚本**

**6\. 性能、稳定性与网络链路分析**

“彻底检查”不仅仅是安全检查，还包括服务的可用性检查。既然这是一个 Web 服务，其响应速度和稳定性至关重要。

### **6.1 合成监控 (Synthetic Monitoring)**

* **UptimeRobot / Pingdom：** 配置这些工具从全球不同节点对 http://120.26.35.49:8080 发起定期（如每 5 分钟）的 HTTP GET 请求 25。  
  * **目的：** 检测服务的稳定性（SLA）。对于开发环境（8080 端口常见用途），服务可能经常重启或崩溃。  
  * **指标：** 关注 HTTP 状态码。频繁的 500（内部错误）或 502（网关错误）表明后端应用存在 Bug 或资源不足。

### **6.2 页面性能分析**

* **GTmetrix / WebPageTest：** 输入 URL 进行测试 27。  
  * **关键指标 \- TTFB (Time To First Byte)：** 首字节时间。如果 8080 端口运行的是动态 Java 应用，高 TTFB（\>500ms）通常意味着数据库查询缓慢或 JVM 堆内存设置不合理（发生频繁 Full GC）。  
  * **瀑布图分析：** 查看静态资源加载情况。如果服务器带宽较小（阿里云 ECS 按带宽计费），大图片可能会阻塞页面加载。

### **6.3 链路质量检测 (MTR)**

由于目标在中国，跨境访问可能不稳定。

* **工具：** mtr (Linux) 或 WinMTR (Windows)。  
* **操作：** mtr \-r 120.26.35.49。  
* **分析：** 观察数据包在每一跳（Hop）的丢包率。如果丢包发生在骨干网节点，说明是网络拥塞；如果丢包发生在最后一跳，说明是服务器负载过高或配置了 ICMP 速率限制。

**7\. 检查结果的解读与后续步骤**

完成上述工具的运行后，通过整合数据可以得出结论。

### **7.1 风险分级处理逻辑**

* **严重（Critical）：** 发现 CVE-2020-1938（Ghostcat）、Struts2 远程代码执行漏洞、Actuator Heapdump 泄露或 Manager 弱口令。  
  * *行动：* 立即切断公网访问，进行应急响应。  
* **高危（High）：** 发现 SQL 注入点、XSS 漏洞、未授权访问的 API 接口。  
  * *行动：* 修复代码逻辑，配置 WAF。  
* **中危（Medium）：** 目录遍历风险、TLS 版本过低（支持 SSLv3/TLS1.0）、缺少安全 HTTP 头（X-Frame-Options）。  
  * *行动：* 优化 Nginx/Tomcat 配置文件。  
* **低危（Low）：** 信息泄露（如 Server 标头显示版本号）、404 页面未自定义。

### **7.2 修复与加固建议**

为了确保 8080 端口的长期安全：

1. **网络隔离（最有效）：** 利用阿里云安全组（Security Group），将 8080 端口的入站规则从 0.0.0.0/0（全网开放）修改为仅允许特定管理 IP 或 VPN 网关 IP 访问。  
2. **中间件升级：** 始终保持 Tomcat、Nginx 或 Spring Boot 为最新稳定版。  
3. **最小权限原则：** 修改 tomcat-users.xml，删除所有默认用户；配置应用以非 Root 用户身份运行。  
4. **关闭不必要的功能：** 禁用 AJP 连接器（注释掉 server.xml 中的 8009 配置）；禁用 Spring Boot Actuator 的敏感端点。

综上所述，对 120.26.35.49:8080 的彻底检查是一个系统工程，需要结合**网络扫描（Nmap）**、**漏洞扫描（Nessus/OpenVAS）**、\*\*Web 渗透（ZAP/Burp）**以及**性能监控（GTmetrix）\*\*等多种工具。只有层层深入，从网络包到应用逻辑逐一验证，才能真正实现“彻底检查好”的目标。
